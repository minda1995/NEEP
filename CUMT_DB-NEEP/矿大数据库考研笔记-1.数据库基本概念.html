    • 1. 数据库四个基本概念
      • 1.1 数据库（DataBase，DB）
        • 所有定义的共同特征：数据库是数据的集合体,能够被计算机管理并为多个用户共享
        • 数据库是指 在计算机的存储设备上合理存放、相关联、有结构的数据集合
          • 数据库首先是指在计算机的存储设备上存放的.属于计算机领域的一个术语
          • 数据库是一个数据集合
          • 这个数据集合是有结构的,这也是和文件系统相比最大的特点之一
          • 这个数据集合是相关联的数据集合,并且只有相关联的数据才可以存放在一起,否则没有意义和研究的价值
          • 这个数据是合理存放的.数据库规范化理论和数据库设计方法专门研究合理存放的问题,这也是数据库技术研究的关键问题之一.
      • 1.2 数据库管理系统（Database Management Syste m，DBMS）
        • 数据库管理系统 是一个操纵和管理数据库的大型软件，由一组计算机程序组成。
        • 建立在用户和操作系统之间的一层数据管理软件
        • 能够对数据库进行有效的管理,包括建立和维护数据库,接受和完成用户访问数据库的各种请求,同时还考虑安全性.完整性.并发控制.故障恢复
        • 主要功能
          • 建立和维护数据库，接受和完成用户访问数据库的各种请求
        • 数据库管理系统结构图
      • 1.3 数据库系统（Database Management System，DBS）
        • 数据库系统是实现有组织地,动态地,存储大量相关的结构化数据,方便用户使用数据库的计算机软件,硬件,资源组成的系统
        • 数据库系统是在计算机系统中引进数据库和数据库管理系统后的组成
        • 数据库系统一般由 硬件、软件、数据库、用户 4部分组成
          • 用户包括：管理人员、开发人员、用户
          • 数据库管理员(DBA)
            • 参与数据库系统的设计和建立
            • 对系统的运行实时监控
            • 定义数据的完整性约束条件
            • 负责数据库性能的改进和数据库的重组及重构工作
      • 1.4 数据库应用系统（Database Application System，DBAS）
        • 由数据库系统及其应用程序组成
        • 在数据库系统环境下建立起来的,为某种应用服务的软硬件的集合
        • 通常被称为 应用软件

    • 2. 数据库系统的产生和发展
      • 2.1 数据、信息、数据管理与数据处理 
        • 数据
          • 用以载荷信息的各种符号
          • 数据本质上是对客观事物特征的一种抽象,符号化的表示
          • 用一定的符号表示那些从观察或测量中所收集的基本事实，采用什么符号完全是人为的规定
        • 信息
          • 数据有意义的表现，信息就是数据的含义
          • 信息实际上就是经过处理后的数据,是消化了的数据
        • 数据和信息联系密切
          • 数据本身不能完全表达自身内容，需要经过解释才能明白其含义，数据和信息的解释是密不可分的
        • 数据管理和数据处理
          • 数据管理技术的好坏,直接影响数据处理的效率
          • 数据处理
            • 把对数据进行收集，组织，存储，加工，抽取和传播等一系列的活动称为 数据处理
            • 数据处理的目的：从大量的原始数据中抽取,推导出对人们有价值的信息
            • 数据处理处理的重点不在计算，而是数据管理
              • 也是数据处理与科学计算的最大差别
          • 数据管理
            • 数据管理是指 对数据的分类、组织、编码、存储、查询和维护等活动
            • 数据管理是数据处理的中心环节
      • 2.2 数据管理技术的产生与发展 #数据库发展三个阶段
        • 数据管理技术经历了 人工管理阶段、文件系统阶段、数据库系统阶段
        • 人工管理阶段
          • 20世纪50年代中期
          • 计算机只有硬件无软件
          • 计算机主要用于科学计算，数据一般不保存在计算机内部
          • 人工管理阶段存在问题：
            • 数据不独立保存
            • 数据不能共享
            • 数据不具有独立性
            • 应用程序管理数据
            • 数据没有软件系统进行管理
              • 程序员不仅要规定数据的逻辑结构
              • 而且还要设计数据的物理结构
              • 数据面向应用
          • 人工管理阶段最明显的缺点：缺乏数据独立性
          • 人工管理阶段.特点：
            • 应用程序管理数据
            • 数据不共享
            • 数据不具有独立性
            • 数据不保存
        • 文件系统阶段
          • 20世纪50年代后期 至 60年代中期
          • 计算机的硬件有了磁盘等直接存储设备，在操作系统中有了专门的数据管理软件-文件系统
          • 采用 应用程序-OS-数据文件 的存取方式
            • 程序与数据可以分别独立存放，数据可以组成数据文件，并且可以独立命名，一旦命名后，程序便可以通过文件名对文件中的数据进行处理，在程序与数据之间需要一个转换过程，这个过程由 文件管理系统 来完成
          • 文件系统
            • 文件应该怎么组织由用户自行决定，每个用户可以建立、维护和处理一个文件或者多个文件。
            • 一个应用程序可以与多个数据文件对应，也可以说一个或多个数据文件为某个应用程序服务
          • 尽管文件系统使得数据管理技术有了重要进展，使得数据可以长期保存，可以由专门的文件系统软件进行数据管理
          • 存在问题:
            • 1. 数据冗余度大、共享性差、易产生数据的不一致性
              • 文件是用户各自建立的，为用户自己或用户组所有，所以即使是相同的数据也必须放在各自的文件中，数据的共享性差，冗余度大
              • 由于相同数据重复存储、各自管理、易产生数据的不一致性
            • 2. 数据独立性差
              • 应用程序与数据之间依赖性很强
              • 数据文件完全是根据具体的应用程序的要求建立的
                • 文件系统中的文件逻辑结构一旦需要修改，那么必须修改应用程序
                • 语言环境的变化要求修改应用程序时，也将引起文件数据结构的改变
            • 3. 用户负担重
              • 文件系统虽然为用户提供了一种简单的、统一的存取文件的方法，但文件的处理、数据的安全性和完整性得不到保证，这些必须由用户程序完成
            • 4. 数据无结构:
              • 由于数据文件是按位置存放的,所以记录之间没有联系，是无结构的
              • 文件系统一般不支持多个应用程序对同一文件的并发访问，数据处理的效率低
              • 使用方式不灵活
                • 每个已经建立的数据文件只限于一定的应用，并且难以对其进行修改和扩充
          • 文件系统阶段.特点：
            • 数据可以长期保存
            • 由文件系统管理数据
            • 数据共享性差
            • 冗余度大
            • 数据独立性差
        • 数据库系统阶段
          • 20世纪60年代 至 now
          • 目标：解决数据独立性问题
            • 克服程序与数据文件相互依赖，力争数据独立
            • 同时还需要尽量解决数据冗余、数据安全性、数据完整性等问题
          • 将过去在文件系统中的 以程序设计为核心，数据服从程序设计 的数据管理模式变为 以数据库设计为核心、应用程序设计退居其次 的数据管理模式
          • 三个里程碑
            • IMS系统-Information Management System
              • 层次数据库
              • 第一个商品化的数据库管理系统，由IBM公司开发
              • 20世纪70年代，在商业金融得到广泛应用
            • DBTG系统-DataBase Task Group
              • 网状数据库
            • 关系数据库系统
          • 数据库系统的三个发展阶段
            • 第一代数据库系统
              • 20世纪70年代以网状型数据库和层次型数据库为代表的。
            • 第二代数据库系统
              • 20世纪80年代以关系数据库为代表的。
            • 新一代数据库系统
              • 20世纪80年代末，90年代初以来热点课题
        • 数据管理三个阶段的比较
      • 2.3 数据库系统特点
        • 数据库系统是一个比较复杂的系统，包括计算机的软件、硬件环境以及数据库、数据库管理系统，包括数据库管理员等相互独立而又相互联系的若干部分组成。
        • 数据库系统管理数据应该具有以下基本特点： #数据库系统特点
          • 数据结构化-采用一定的数据模型实现数据结构化
            • 数据结构化 是 数据库系统与文件系统的根本区别之一
              • 文件系统中，在整体上不存在结构化，并且数据的存放只有程序员了解数据的含义、存放的位置等
              • 数据库系统中的数据是按一定的数据模型来组织、描述和存储的
                • 数据模型能够表示现实世界中各种数据组织结构和数据间的联系
            • 数据库是从整体上考虑数据结构，所以数据不再是面向应用而是面向系统。
              • 对于不同的应用，可以选取整体模型的各种合理子集加以实现
          • 数据冗余度小、共享性高，避免了数据的不一致性
            • 相同的数据在数据库中一般只存储一次，并为不同的应用共享，大大降低了数据的冗余度，提高了共享性
            • 一个数据在数据库中只存储了一次（无冗余数据），不会产生数据的不一致问题
          • 具有较高的数据独立性 #数据独立性
            • 数据独立性 是指 数据库中的数据与应用程序的无关性
            • 逻辑独立性
              • 数据的 全局逻辑结构 和 局部逻辑结构 之间相互独立
              • 当全局逻辑结构改变时，可以改变全局逻辑结构与局部逻辑结构之间的映射，使得与某个具体应用相关的局部逻辑结构可以不改变，从而不用改变应用程序
            • 物理独立性
              • 数据 存储结构 与 全局逻辑结构 之间的相互独立性
              • 当改变数据库中的存储结构时，不影响全局逻辑结构，只要不改变全局逻辑结构，应用程序不需要改变
          • 数据由DBMS统一管理和控制
            • 文件系统与数据库系统的对比：
              • 在文件系统中，虽然数据由操作系统中的文件系统管理，但缺乏安全性、完整性等控制
              • 数据库系统由数据库管理员（DBA）对数据库进行管理和维护
            • 数据库系统提供的数据控制功能：
              • 数据的安全性保护
                • 数据安全性 是指 保护数据以防止不合法地使用造成数据的泄密和破坏
                • 保护措施
                  • 通常采取 用户识别与鉴别 和 存取控制措施 实现安全保护
                  • 用户标识与鉴别
                    • 每次用户要求进入系统时，由系统进行核对，只有合法用户才可以具有使用权。
                  • 存取控制措施
                    • 确保只授权给有资格的用户访问数据库的权限
                    • 权限分配：一般数据库用户、数据库拥有者、有DBA权限的用户
                    • 不同权限的用户由于权限不同，所访问到的数据库中的数据不同，从而起到安全保护的作用
              • 数据的完整性保护
                • 数据的完整性 是指 将数据控制在有效范围内或使数据之间满足一定的关系，以保证数据的正确性、有效性和相容性
                • 即 为了防止数据库中存在不符合语义的数据，防止错误信息的输入和输出
              • 并发控制
                • 如果对并发操作不加控制，会带来 数据丢失修改 的问题，还会带来 脏数据 和 数据不可重复读
              • 数据库恢复
                • 当数据库系统发生故障时，DBMS必须具有将 数据库从错误状态恢复到某个正确状态的功能
                • 数据库恢复机制 是为保证 事务的原子性和持久性
                  • 当数据库发生故障时，有些事务没有完成就被迫停止，这些未完成的事务所做的操作可能已经对数据库造成影响，使得数据库处于不一致的状态。所以需要DBMS的恢复机制撤销所有未完成事务对数据库的一切影响，保证事务的原子性；同样，对于已经提交的事务要恢复他对数据的更改，保证事务的持久性
                  • DBMS所采用的恢复技术是否行之有效，不仅对系统的可靠程度起着决定作用，而且对于系统的运行效率也有很大影响，是衡量系统性能优劣的重要指标
                • 数据库系统中可能发生的故障很多，如 事务故障、系统故障和介质故障
                • 恢复数据库的基本原理很简单：数据库中的任何一部分被破坏的或不正确的数据可以根据存储在系统别处的冗余数据来重建
                  • 最常用的冗余数据有后备副本和日志文件
                  • 尽管恢复的基本原理很简单，但实现的细节却相当复杂
    • 3. 数据模型 #数据模型
      • 3.1 数据模型的概述
        • 数据模型的定义
          • 数据库技术中的数据模型是指 现实世界数据和信息的模拟和抽象
          • 数据模型是数据库系统的核心和基础，现有的数据库系统均是基于某种数据模型的
        • 数据模型应该满足的要求
          • 现实世界到计算机世界的抽象过程：
            • 1.对现实世界的事物进行分析，抽象成 概念数据模型，然后将概念模型转化为计算机进行处理的逻辑数据模型，最后转化为计算机能实现的存储模型
            • [现实世界到计算机世界的抽象过程]
          • 无论是数据库设计还是DBMS的实现，都需要数据模型，以达到将现实系统向计算机化管理转变的目标
          • 必须具备以下要求：
            • 真实性
              • 要求能够真实地模拟现实世界，也只有真实地反映出现实世界所对应的要求才是有意义的
            • 易理解性
              • 对现实世界模拟以后，可以抽象出概念模型
              • 对于概念模型所描述的是否正确，需要和现实世界的用户进行沟通，如果概念模型用户不理解，就无法沟通，要求概念模型具有易理解性。同时，概念模型还要向数据模型转换，不过不具备易理解性，设计者也会出现困惑
            • 易实现性
              • 易实现性主要针对概念模型和数据模型，如何能够使得概念模型便于转化为数据模型，同伙私数据模型又方便在计算机上实现
        • 数据模型的分类
          • 按照不同的应用层次，数据模型可以分成3中类型：概念数据模型、逻辑数据模型、物理数据模型
          • 概念数据模型：概念模型
            • 独立于计算机系统的数据模型，完全不涉及信息在计算机中的表示，是面向数据库用户的现实世界模型，主要用来描述现实世界的概念化结构
            • 摆脱具体技术问题，集中于分析数据和数据之间的联系
            • 最常用的 E-R模型
          • 逻辑数据模型：逻辑模型
            • 用户从数据库看到的模型，是具体的DBMS所支持的数据模型
            • 主要有：层次模型、网状模型、关系模型、面向对象模型
            • 既面向用户又面向系统，主要用于DBMS的实现
          • 物理数据模型：物理模型
            • 面向计算机系统的，最底层的抽象
            • 描述数据在系统内部的表示方式和存取方式
            • 不但与具体的DBMS有关，还与操作系统和硬件有关
            • DBMS为了保证其独立性与可移植性，大部分物理数据模型的实现工作由系统自动完成，设计者只设计索引、聚集等特殊结构
        • 数据模型的组成要素
          • 一个数据库的数据模型应包含：数据结构、数据操作、数据完整性约束
            • 其中数据结构是描述一个模型性质的最重要的方面
          • 数据结构-是对系统静态特性的描述
            • 是对实体模型与实体间联系的表达和实现
            • 规定了如何描述数据的类型、内容、性质和数据之间的相互关系
            • 数据模型的 最基本组成部分
            • 规定了 数据模型的 静态特征
            • 是描述一个模型性质的最重要的方面
          • 数据操作-是对系统静态特性的描述
            • 一组用于指定数据结构的任何有效实例执行的操作或推导规则
            • 主要的操作：查询和更新（插入、修改、删除）
              • 数据模型要为这些操作定义确切的含义、操作规则和实现操作的语言
            • 规定了 数据模型的 动态特征
          • 数据完整性约束-一组完整性规则的集合
            • 给出数据及其联系应具有的制约和依赖规则
              • 定义了给数据模型中数据及其联系所具有的制约和依存规则，用以限定相容的数据库状态的集合和可容许的状态改变
            • 保证数据库中数据的正确性、有效性、相容性
      • 3.2 实体联系数据模型
        • 概念模型是从现实世界到数据世界的一个中间层次，是数据库设计的重要工具
        • 特点
          • 具有丰富的语义表达能力和直接模拟现实世界的能力，具有直观、自然、语义丰富、易于用户理解
          • 具有丰富的语义表达能力和直接模拟现实世界的能力
        • 概念模型必须能够真实反映现实世界中被管理事物的特征及其复杂的联系
          • 即 应具有丰富的语义表达能力和直接模拟现实世界的能力，且具有 直观、自然、语言丰富、易于用户理解的特点
        • 目前广泛使用的概念模型是 E-R模型，即 实体-联系数据模型
        • 信息的三个领域
          • 现实世界
            • 研究分析现实世界事物的规律和特点是建立概念模型的基础
            • 实体
              • 客观存在并可相互区别的个体
            • 实体特征
              • 任何一个实体都具有它自己的特征和特质，描述实体的主要特征的这一特性称为 实体特征
            • 实体集
              • 具有相同实体特征的这一类实体的集合称为实体集
            • 实体标识符
              • 实体集中唯一能确定实体集中某个实体的最小实体特性集
          • 信息世界
            • 就是现实世界在人头脑中的反映
            • 在信息世界中
              • 用实体记录表示实体
              • 用实体记录集表示实体集
              • 属性表示实体集的特征
              • 标识属性表示一个实体标识符
            • 属性 的几点说明：
              • 一个实体记录可以有多个属性，在研究某一实体记录时，关心的是那些感兴趣的属性
              • 不能再细分的属性称为 原子属性；可以再细分的属性称为 可分属性
                • 原子属性与可分属性之间是相对的
                • 在实际情况中，要根据具体问题来使用属性概念
              • 属性有“型”和“值”之分，属性的具体表现称为属性值
                • 属性值是附加于属性的，有什么样的属性，就有什么样的属性值
                • 属性的型是相对稳定的，属性的值随着属性的型相对变化
              • 每个属性值都有一定的变化范围
          • 计算机世界

            • 记录：信息世界中的 实体记录 在计算机世界中的表示

      • 3.3 数据结构模型常用结构
        • 常用的数据模型结构有4种：层次模型、网状模型、关系模型、面向对象模型
        • 层次模型
          • 数据结构
            • 按照 树形结构 来表示实体集与实体集之间的联系模型。
              • 层次模型中用记录类型描述表示实体集，每个节点表示一个记录类型
              • 节点之间联系的基本方式 1：n
            • 满足条件：
              • 有且只有一个节点没有双亲节点，称为 根节点
              • 除根节点以外的节点有且仅有一个双亲节点
            • 层次模型例子
          • 数据操作
            • 1）查询：
              • 从根结点开始，按给定条件沿一个层次路径查找所需要的记录。
            • 2）更新：
              • ①插入：指定一个插入层次路径，完成数据的插入操作 。
              • ②删除：先定位到要删除的记录上，完成删除任务。
              • ③修改：先定位，然后可将修改后的记录值写回到数据库中。
          • 数据约束
          • 层次模型的特点
            • 记录之间的联系通过 指针 来实现
            • 常用的物理实现方法有 邻接法 和 链接法
              • 邻接法
                • 用连续的物理顺序表示记录之间联系的方法，在该方法中，由根记录开始存放，按照自顶向下、从左到右的顺序记录
              • 链接法
                • 采用指针来实现记录之间联系的方法
                • 用指针按层次顺序把各个记录链接起来，而各个记录存储时不一定按层次顺序
            • 优点-简单清晰、性能较高、良好的完整性支持
              • 查询效率高
              • 结构简单、层次分明、便于在计算机内实现
              • 层次间路径唯一 ，为有效地进行数据操作也提供了条件，拥有较好的完整性
            • 缺点-不能直接表示多对多关系、操作限制多、结构严密，层次命令程序化
              • 缺乏直接表达现实世界中非层次结构的复杂联系.如,多对多
                • 虽然有解决方法，用户不容易掌握
              • 层次顺序的严格限制,使得对插入或删除操作也带来较多的限制
                • 查询子节点必须通过双亲节点，使得应用程序的编写比较复杂
            • 、结构严密，层次命令程序化
        • 网状模型
          • 满足条件
            • 允许一个以上的结点无双亲
            • 一个结点可以有多于一个的双亲
          • 表示方法
            • 在网状模型中，用有向图结构表示实体类型及实体间的联系，有向图中的每个节点表示一个实体集，称之为记录型，用矩形框表示。
            • 记录间的联系是通过“系”实现的，系用有向线段表示，箭头指向 1:n 的n方
          • 特点
            • 网状数据模型是一种比层次数据模型更具普遍性的结构，反映了实体集间普遍存在更为复杂的联系
              • 层次结构实际上是网状结构的一个特例
            • 优点:
              • 能直接描述现实世界
              • 查询效率高，记录之间的联系通过指针
            • 缺点
              • 结构复杂，难掌握
              • 其DDL，DML语言复杂，不易使用
        • 关系模型
          • 用二维表格表示现实世界实体集及实体集间的联系
            • 在数学上有严格定义，数据结构简单，容易被初学者理解
          • 关系模型中
            • 二维表称为关系，表中的列成为属性，类中的值取自相应的域，域是属性所有可能取值的集合。
            • 表中的一行称为元组，元组用关键字标识，每项称为分量。
            • 对二维表框架的描述称为关系模式
          • 每个关系都是由同一个记录构成，不同的关系可以有相同的属性，他表示关系间的联系，实体集之间的联系是通过在二维表中存放两个实体集的关键字实现的
          • 关系模式的描述功能表明：
            • 无论是对现实世界实体集的描述，还是对实体集之间联系的描述都可以采用统一的数据结构
          • 关系的性质：
            • 关系中的每个属性是不可分解的, 即所有域都是原子数据的集合
            • 没有完全相同的行或列, 行和列的排列顺序无关紧要
          • 关系模式
            • 关系模式是关系中信息内容结构的描述
            • 关系模式包含 关系名, 属性名, 每个属性列的取值, 数据完整性约束条件以及各个属性间固有的数据依赖关系
            • 关系模式
          • 联系的实现
            • 在层次和网状模型中，联系使用指针来实现的，在关系模型中，联系通过关系模式的关键字来实现
            • 关系模式与层次、网状最大的不同，即使用关键字，而不是指针
          • 数据操作-在关系数据库中，对数据库的查询和更新操作都归结为对关系的运算
            • 以一个或多个关系为运算对象，对他们进行某些运算，形成一个新的关系，提供用户所需数据
            • 关系模型中的关系代数和关系演算为关系运算提供了丰富的操作功能
            • 分为关系代数和关系演算
              • 关系代数
                • 传统的集合运算
                • 特殊的关系运算
              • 关系演算
                • 以数理逻辑中的谓词演算来表达关系的操作
          • 关系模型的数据约束
            • 实体完整性
              • 实体完整性约束是指任一关系中标识属性（关键字）的值不能为NULL
            • 参照完整性
              • 不同关系之间的约束
                • 当存在关系间的引用时，要求不能引用不存在的元组
              • F是R(U)的外关键字，并是S(U)的关键字，则F属性的值必须满足：
                • 1.或者取空值(NULL)
                • 2.或者等于S(U)中某个元组的关键字值
            • 用户自定义完整性
          • 关系模型主要优点-坚实的理论基础、表达能力强、简单、数据独立性高
            • 关系模型有坚实的理论基础
              • 关系模型是唯一可数学化的模型
            • 二维表不仅能表示实体集，而且能方便地标识实体集间的联系，有很强的表达能力
            • 简单，易学易用
              • 数据的表示方法统一、简单，便于在计算机中实现
              • 向用户提供的是简单的关系模型，不涉及系统内部复杂的联系
            • 数据独立性高
              • 为关系数据库的建立、扩充、维护、调整和重构提供了方便
          • 关系模式的缺点-效率低
            • 查询效率不如非关系模型
              • 由于存取路径对用户透明，查询优化处理依靠系统完成，加重系统的负担
        • 面向对象数据模型
          • 面向对象数据模型 是面向对象程序设计方法和数据库技术相结合的产物
            • 基本目标：以更加接近人类思维的方式描述客观世界的事务以及其联系，且使描述问题的问题空间和解决问题的方法空间在结构上尽可能一致，以便对客观实体进行结构模拟和行为模拟
          • 将现实世界的一切事物看作对象，一个对象不仅包括描述它的数据，而且还包括对它进行操作方法的定义
            • 也可以将面向对象数据模型看作是一种可扩充的数据结构
          • 这种数据模型比传统数据模型有更丰富的语义，用户根据应用需要可以定义新的数据类型及相应的约束和操作
          • 面向对象数据模型中 最基本的概念是 对象 和 类
            • 对象
              • 对象与实体
                • 将所有现实世界的任一实体都模拟为一个对象。
                • 对象是客观存在并能互相区别
                • 对象是客观世界中概念化的基本实体
                  • 因此实体都可以是为对象
                  • 复杂对象可以由相对简单的对象以某种方法组合而成
              • 对象与记录
                • 对象与记录的概念相似，但比记录更为复杂
              • 对象的属性与方法
                • 一个对象包含若干属性，用来描述对象的状态、组成和特性
                  • 在面向对象模型中，允许属性也是对象，还可以包含其他对象作为其属性
                • 对象还包含若干方法，用以描述对象的行为特征
                  • 方法的定义和表示包括两部分
                    • 方法的接口
                      • 给出方法的外部表示、包含的名称、参数及结果类型
                    • 方法的实现
                      • 用程序设计语言编写的一段可以对对象进行某种操作的程序，用以实现方法的功能
                    • 方法也称为操作，可以改变对象的状态
                • 系统把对一个对象的属性和方法封装为一个整体
                  • 对象内部对用户的隐蔽的
                  • 封装的目的：
                    • 为了使对象的使用和实现分开，用户不必了解内部操作实现的细节，只需用发送消息的方法实现对对象的访问
                  • 从程序设计看，对象是一个程序模块；从用户看，对象提供了所期望的行为
                  • 这种使数据和操作封装在一起的建模方法，有利于程序的模块化，可以提高系统的可维护性和易修改性
                • 对象的通信
                  • 为了实现对象与对象之间的联系，对象之间要进行通信，把通信的构造称为消息
                  • 消息是用来请求对象执行某一操作或回答某些信息的要求
                  • 消息传递的过程
                    • 当一个消息发送给某个对象时
                      • 消息中包含要求接收对象去执行某些活动的信息
                      • 接收到信息的对象经过解释进行响应的操作
                      • 并以消息的形式返回操作结果
                  • 消息一般由三个部分组成
                    • 接收者
                      • 消息所施加作用的对象
                    • 操作者
                      • 给出消息的操作要求
                    • 操作参数
                      • 消息操作时所需要的外部数据
            • 类和类的实例
              • 类是具有共同属性和方法的对象的集合
                • 这些属性和方法可以在类中统一说明，而不必在类的每个实例中重复说明
              • 类中的每个对象称为该类的一个实例
                • 同一类中对象的属性名虽然相同，当这些属性的取值会因各个实例而差异，类的属性 实际上是个变量
              • 在面向对象模型中，类被组织成一个有根的有向无环图，称为 类层次结构
              • 面向对象模型的特点
                • 具有封装性、信息隐匿性
                • 持久性、继承性、代码共享、软件重用性
              • 面向对象模型最大的特点：
                • 有丰富的语义便于更自然的描述现实世界

    • 4. 数据库系统结构
      • 数据库系统的体系结构
        • 从数据库管理系统的抽象层次看，数据库通常采用三级模式结构；
        • 从数据库最终用户角度看，数据库系统结构可以分为 集中式数据库结构、网络环境下的客户\服务器结构、分布式数据结构系统、并行数据库系统结构
      • 4.1 数据库系统的三级模式结构
        • 一个数据库结构从逻辑上可以划分为三个层次：外模式、概念模式、内模式
        • 三级模式仅仅是对数据的描述, 而数据实际上只存在于物理层
        • [三级模式结构图解]
        • 说明：
          • 型：对某一类数据的结构和属性的说明
          • 值：是型的一个具体赋值
        • 概念模式-模式
          • 概念模式简称 模式，是对数据库 全局逻辑 的描述，反映的是数据的结构及其联系
          • 模式是对 型 的描述，反映的是数据的结构及其联系，模式是相对稳定的
          • 数据库只有一个概念模式
            • 是数据库系统三级模式结构的中间层
            • 不涉及数据的物理存储细节和硬件环境
            • 与具体点应用程序及程序设计语言无关
          • 概念模式由DBMS提供的模式描述语言（模式DDL）来定义和描述
        • 外模式
          • 外模式简称 子模式
            • 是数据库用户能够看见和使用的局部数据的逻辑结构和特征的描述
            • 介于模式和用户之间
              • 是用户与数据库之间的接口
          • 模式与外模式
            • 外模式是概念模式的一部分，是一个逻辑子集
            • 外模式表示 实体、实体属性和实体间的联系
            • 数据库可以有多个外模式
              • 同一外模式可以为某一用户的多个应用系统
              • 但一个应用程序只能使用一个外部模式
              • 外模式与应用的关系是一对多
            • 模式与外模式是一对多的关系
          • 外模式由DBMS提供的外部模式描述语言（外部模式DDL）来定义和描述
          • 优点
            • 不仅可以使用户对数据库操作方便，而且有利于数据共享
              • 从同一模式中产生若干个外模式供若干个用户使用
              • 不仅实现了 数据共享，还减少了数据冗余，减少了数据潜在的不一致性
            • 由于外模式与模式间的映射关系保证了数据的独立性
            • 由于用户只能使用在给定的外模式定义范围内的数据，确保了其他数据的安全
        • 内模式
          • 内模式也称 存储模式
            • 是对数据库中数据 物理结构 和 存储方式 的描述
            • 是数据在数据库内部的表示方式
          • 数据库只有一个内模式
            • 内模式中规定了 所有的内部记录类型、索引和文件的组织方式，以及数据控制方面的细节
          • 内模式是物理设备上存储数据的物理抽象
          • 内模式由DBMS提供的内部模式描述语言（内部模式DDL）来定义和描述
      • 4.2 数据库的两级映像和数据独立性 #两级映像 #数据独立性
        • 数据库系统的是三级模式是对数据进行3个级别的抽象。
          • 把数据的具体组织留给DBMS，用户只要抽象处理数据，而不必关心数据在机器中的具体表示方式和存储方式
        • 两级映像 #两级映像
          • 这两层映象保证了数据库系统中的数据具有较高的逻辑独立性和物理独立性
          • 映像
            • 指出了映像双方是如何进行转换的
          • DBMS在三级模式间提供了 两级映像
            • 数据库的三级结构是靠映像来联系和转换的
          • 数据库的两级映像保障了数据的独立性
            • 数据独立性
              • 在三层模式中的定义：数据库系统在某一层次模式上的改变不会使它的上一层模式也发生改变的能力
              • 逻辑独立性
                • 用户数据独立于全局逻辑数据的特性称为 逻辑独立性
                • 一旦模式发生变化，无需改变外模式或是应用程序的能力
              • 物理独立性
                • 全局数据独立于物理数据的特定称为 物理独立性
                • 不会因为内模式发生改变而导致概念模式发生改变的能力
          • 外模式|模式 映像
            • 定义了各个外模式与模式之间的映像关系
            • 对应于同一个模式可以有多个外模式，对于每个外模式，数据库系统都有一个 外模式|模式 映像
              • 定义了该外模式与模式之间的对应关系
            • 数据库的模式改变，只要对 外模式|模式映像进行相应的修改，可以使得外模式和应用程序保持不变
              • 保证了数据与程序的 逻辑独立性
          • 模式|内模式 映像
            • 定义了数据库全局逻辑与存储结构之间的对应关系
            • 数据库的内模式（物理存储结构）发生改变，可对 模式|内模式映像做修改，使得模式尽量保持不变，从而不必修改应用程序
              • 保证了数据与程序的 物理独立性
        • 数据库系统具有数据独立性，数据库系统把用户数据与物理数据完全分开，是用户拜托了繁琐的物理存储细节，减少了应用程序维护的开销
        • 数据库的3个抽象层次
          • 数据库的三级模式结构定义了数据库的3个抽象层次：
            • 物理数据库
              • 以内模式为框架的数据库
              • 是数据库中最里面的一个层次，是物理存储设备上实际存储着的数据集合
              • 这些数据称为 原始数据，是用户处理的对象
              • 系统程序员视图
                • 数据的存储结构
            • 概念数据库

              • 数据库结构中的一个中间层次，数据库的整体逻辑表示
                • 描述了每个数据的逻辑定义及数据间的逻辑联系
              • 概念数据库描述里数据库系统所有对象的逻辑关系，不涉及他们的物理存储情况
              • 概念数据库本身并不是一个实际存在的数据库，是实际存在的物理数据库的一种逻辑描述
              • 是DBA概念下的数据库
            • 逻辑数据库
              • 以外模式为框架的数据库
              • 是数据库结构的最外一层，是用户所看到和使用的数据库
              • 逻辑数据库是某个或某些用户使用的数据集合
              • 用户根据系统提供的外模式用查询语言或应用程序对数据库的数据进行操作

    • 1.1 名词解释
      • 数据库 (DB)：在计算机的存储设备上合理存放、相关联、有结构的数据集合。
      • 数据库系统 (DBS)：在计算机系统中引进数据库和数据库管理系统后的组成。或者是实现有组织地、动态地存储大量相关的结构化数据，方便用户使用数据库的计算机软件、硬件资源组成的系统。
      • 数据库管理系统 (DBMS)：一个操纵和管理数据库的大型软件，它由一组计算机程序构成。它是位于用户与操作系统之间的一层数据管理软件。
      • 实体：客观存在并可相互区别的个体。
      • 实体集：具有相同实体特性实体的集合。
      • 属性：表示实体的特性。
      • 关键字：能够唯一标识文件中的某一个记录的最小字段集。
      • 数据模型：现实世界数据与信息的模拟和抽象。
        • 概念数据模型：简称概念模型，是独立于计算机系统的数据模型，完全不涉及信息在计算机中的表示，是面向数据库用户的现实世界模型，主要用来描述现实世界的概念化结构。
        • 关系模型：是最重要的一种数据模型，也是目前主要采用的数据模型，用二维表格表示现实世界实体集及实体集间的联系。
      • 模式：是由数据库设计者综合所有用户数据，按照统一的观点构造的对数据库全局逻辑结构的描述。
        • 外模式：介于模式与应用之间，是用户与数据库之间的接口，是数据库用户（包括应用程序员和最终用户）能够看见和使用的局部数据的逻辑结构和特征的描述。
        • 内模式：也称存储模式，是对数据库中数据物理结构和存储方式的描述，是数据在数据库内部的表示方式。
      • 数据独立性：应用程序和数据结构之间相互独立，不受影响。
        • 逻辑数据独立性：表示一旦模式发生变化，无需改变外模式或应用程序的能力。
        • 物理数据独立性：表示不会因为内模式发生改变而导致概念模式发生改变的能力。
      • 一对一联系：设有两个实体集 A、 B，如果对于实体集 A 中的每一个实体， B 中至多有一个实体与之有联系，反之亦然，则称 A、 B 有一对一联系（ 1： 1 联系） 。
      • 一对多联系：设有两个实体集 A、 B，若 A 中每个实体与 B中任意个实体（包括零个）相联系，而 B 中每个实体至多和 A 中一个实体有联系，则称 A 和 B 是一对多联系（ 1： n 联系） 。
      • 多对多联系：设有两个实体集 A、 B，若两个实体集 A、 B 中的每一个实体都和另一个实体集中任意个实体（包括零个）有联系，则称 A、 B是多对多联系（ m： n 联系） 。

      • 2.数据库系统的三个发展阶段是什么？
        • 1) 层次型和网状型：代表产品是 1969 年 IBM公司研制的层次模型数据库管理系统 IMS。
        • 2) 关系型数据型库：目前大部分数据库采用的是关系型数据库。 1970 年 IBM 公司的研究员 E.F.Codd 提出了关系模型。典型的关系 DBMS产品有 Oracle 、 Sybase、 SQLServer 、 DB2、 Access 、 VFP等等。
        • 3) 第三代数据库将为更加丰富的数据模型和更强大的数据管理功能为特征， 以提供传统数据库系统难以支持的新应用。 它必须支持面向对象， 具有开放性， 能够在多个平台上使用。
      • 3.文件系统与数据库系统有何区别和联系？
        • 文件系统和数据库系统之间的区别：
          • 1） 文件系统用文件将数据长期保存在外存上，数据库系统用数据库统一存储数据；
          • 2） 文件系统中的程序和数据有一定的联系， 数据库系统中的程序和数据分离， 数据具有较高的独立性；
          • 3） 文件系统用操作系统中的存取方法对数据进行管理，数据库系统用 DBMS统一管理和控制数据；
          • 4） 文件系统实现以文件为单位的数据共享， 数据库系统实现以记录和字段为单位的数据共享，数据结构化，冗余度小。
        • 文件系统和数据库系统之间的联系：
          • 1） 均为数据组织的管理技术；
          • 2） 均由数据管理软件管理数据，程序与数据之间用存取方法进行转换；
          • 3） 数据库系统是在文件系统的基础上发展而来的。
      • 4.数据库系统的特点。
        • 1) 采用一定的数据模型实现数据结构化
        • 2) 数据共享、冗余度小、避免了数据的不一致
        • 3) 具有较高的数据独立性4) 数据由 DBMS 统一管理和控制

      • 10.层次数据模型、网状数据模型和关系数据模型的优点和缺点。
        • 1）层次模型具有查询效率较高、结构简单、层次分明，便于在计算机内实现的优点，在层次数据结构中， 从根结点到树中任意结点均存在一条唯一的层次路径， 为有效地进行数据操纵也提供了条件，并且层次数据模型提供了良好的完整性支持。但层次模型也有缺点， 一是层次数据模型缺乏直接表达现实世界中非层次型结构的复杂联系， 二是层次顺序的严格限制， 使得对插入或删除操作也带来了较多的限制， 并且查询子女结点必须通过双亲结点，所以使得应用程序的编写也比较复杂。

      • 12.什么是外部模式？概念模式？内部模式？它们之间有何联系？这种分级结构的优点是什么？
        • 外部模式又称子模式， 介于模式与应用之间， 是用户与数据库之间的接口， 是数据库用户（包括应用程序员和最终用户）能够看见和使用的局部数据的逻辑结构和特征的描述。
        • 概念模式简称模式， 是由数据库设计者综合所有用户数据， 按照统一的观点构造的对数据库全局逻辑结构的描述。
        • 内部模式也称存储模式， 是对数据库中数据物理结构和存储方式的描述， 是数据在数据库内部的表示方式。
        • 一个数据库结构从逻辑上可以划分为三个层次：外部模式（ External Schema），概念模式（Conceptual Schema）和内部模式（ Internal Schema），称为数据库系统的三级模式结构。数据库系统的三级模式是对数据进行三个级别的抽象。 它把数据的具体组织留给 DBMS去做， 用户只要抽象地处理数据， 而不必关心数据在机器中的具体表示方式和存储方式。 数据库的三级结构是依靠映像来联系和互相转换的。 正是这两层映像保证了数据库系统中的数据具有较高的数据独立性。

        • 1) 数据独立性： 数据库系统在某一层次模式上的改变不会使它的上一层模式也发生改变的过程。数据独立性包括数据的逻辑独立性和物理独立性两种。
        • 2) 一个数据库结构从逻辑上可以划分为三个层次：外部模式（ External Schema），概念模式（Conceptual Schema）和内部模式（ Internal Schema），称为数据库系统的三级模式结构。
        • 数据库的三级结构是依靠外模式 / 模式映像和模式 / 内模式映像来联系和互相转换的。 正是这两层映像保证了数据库系统中的数据具有较高的数据独立性，其中外模式 / 模式映像保证逻辑独立性，模式 / 内模式映像保证物理独立性。
      • 14.数据库系统的软件包括哪些主要成份？
        • 基本的系统软件是操作系统， 其它任何软件都必须在它的支持下工作。
        • 除操作系统之外，还必须配有数据库管理系统， 没有数据库管理系统也就不能称其为是数据库系统。
        • 同时， 为了开发数据库应用系统，还需要有各种高级语言及其编译系统。
      • 15.什么是数据字典？它在数据库中的作用是什么？
        • 数据字典（ Data Dictionary ）是描述各级模式的信息，主要包括所有数据的结构名、意义、描述定义、存储格式、完整性约束、使用权限等信息。由于数据字典包含了数据库系统中的大量描述信息（而不是用户数据） ，因此也称它为“描述信息库” 。
        • 数据字典是 DBMS 存取和管理数据的基本依据，主要由系统管理和使用
      • 16. DBA的主要职责是什么？
        • ①参与数据库系统的设计与建立。
        • ②对系统的运行实行监控。
        • ③定义数据的安全性要求和完整性约束条件。
        • ④负责数据库性能的改进和数据库的重组及重构工作。
      • 17.用户访问数据库的过程。
        • 1) 接受应用程序的数据请求
        • 2)DBMS对用户的操作请求进行分析
        • 3) 数据库管理系统要向操作系统发出操作请求
        • 4) 操作系统接到命令后， 对数据库中的数据进行处理， 将结果送到系统缓冲区， 并发出读完标志。
        • 5)DBMS接到回答信号后，将缓冲区的数据经过模式映射，变成用户的逻辑记录送到用户工作区，同时给用户回答成功与否的信息。
      • 18.关系数据库完整性约束有哪些？举例说明。
        • 1) 实体完整性约束： 实体完整性约束是指任一关系中标识属性 （关键字）的值，不能为NULL，否则，无法识别关系中的元组。
        • 2) 参照完整性约束： 参照完整性是不同关系间的一种约束， 当存在关系间的引用时， 要求不能引用不存在的元组。若属性组 F 是关系 R（ U）的外关键字，并是关系 S（U）的关键字( 即 F 不是 R(U)的关键字， 而是 S(U) 的关键字， 称 F 是 R （ U） 的外关键字 )， 则对于 R（ U）中的每个元组在属性组 F 上的值必须为：空值（ NULL ）或者等于 S（ U）中某个元组的关键字值。
        • 3）用户定义完整性约束。如值的类型、宽度等。
      • 19.有控制的冗余与无控制的冗余有什么区别？
        • 有控制的冗余是指数据库设计者在设计中为了提高操作效率有意保留的冗余数据
        • 无控制冗余是在数据库设计中由于属性间的依赖而存在的关联数据
        • 两者的区别是无控制冗余不是数据库设计者所要保留的冗余数据， 因为它会带来很多问题， 使得存储效率比较低， 而有控制冗余是数据库设计者有意保留的数据，主要是为了改善对数据库的查询效率。
      • 20.简述数据库、数据库管理系统和数据库系统之间的关系？
        • 数据库是指在计算机的存储设备上合理存放、 相关联、 有结构的数据集合。 这个数据的集合由数据库管理系统进行建立和维护， 用户访问数据库中的各种请求都是通过数据库管理系统完成的。
        • 数据库系统是在计算机系统中引进数据库和数据库管理系统后的组成， 没有数据库和数据库管理系统不能够构成数据库系统。
