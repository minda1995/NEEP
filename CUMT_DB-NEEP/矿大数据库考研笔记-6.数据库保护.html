矿大数据库考研笔记-6.数据库保护
  • 考试范围及要求
    • 数据库完整性
    • 数据库安全性
    • 数据库并发控制
    • 数据库恢复技术
  • 书本知识梳理
    • 前言
      • 数据库
        • 数据库是按照某种数据模型组织并存储在一起的相关数据集合
        • 最大的特点是 实现数据共享
        • 为了保证数据共享的安全性，需要对数据库实施保护
      • 主要的保护工作有 4个方面:
        • 备份和恢复
          • 保证数据库发生故障时能够恢复到正确的状态
        • 并发控制
          • 保证多用户并发访问数据时操作的正确性
        • 完整性控制
          • 保证数据的正确性, 有效性和一致性
        • 安全性控制
          • 防止数据丢失, 错误更新, 越权使用等问题
    • 1. 事务
      • 1.1 事务的概念
        • 事务
          • 是用户定义的一个数据库操作序列
          • 这些操作要么全做、要么全不做，是一个不可分割的工作单位
          • 在关系数据库中一个事务可以是一条SQL语句, 一组SQL语句或整个程序
        • 定义事务的语句:
          • BEGIN TRANSACTION /开始一个事务
          • COMMIT /提交事务
          • ROLLBACK /撤销事务
        • 事务的开始与结束
          • 隐式控制
            • 由DBMS按缺省规定自动划分。
          • 显式控制
            • BEGIN   TRANSACTION
            • [事务开始]
            • COMMIT
            • [事务提交，重新改写数据库]
            • ROLLBACK
            • [事务提交，发生错误撤消]
          • 通常事务以 BEGIN TRANSACTION 开始，标志事务开始执行，并以COMMIT 或 ROLLBACK 结束
          • COMMIT
            • COMMIT 表示提交，即提交事务的所有操作
            • 事务中所有对数据库的更新写回到磁盘上的物理数据库中，事务正常结束
          • ROLLBACK
            • ROLLBACK 表示回滚，即事务没有成功结束。
            • 出现错误，必须撤销事务对数据库所做的任何改变和影响，回滚到事务开始前的数据库状态
        • 例子
          • [commit]
            • BEGIN   TRANSACTION
              • INSERT
              • INTO  S(S#, Sname, Sage, Sdept)
              • VALUES(‘10002’, ’李娜’, 18,‘计算机’)
            • COMMIT
          • [rollback]
            • BEGIN   TRANSACTION
              • DELETE
              • FROM      S
              • WHERE     S#=‘10002’;
              • DELETE
              • FROM      SC
              • WHERE     S#=‘10002’;
            • ROLLBACK
      • 1.2 事务的特性
        • 作为不可分割的工作单位，事务有4个特性，包括 原子性、一致性、隔离性、持续性。
          • 这四个基本特性刻画出了事务的本质，简称ACID 特性
        • 数据库并发控制和恢复机制的目标
          • 保证在事务并发执行时或在系统发生故障时仍然满足事务的ACID属性
        • 1. 原子性-【事务是不可分割的工作单位】
          • 事务是处理的一个原子单位，全部执行或者都不执行
          • 因故障没有全部完成，该事务操作被认为是无效的，要故障恢复时消除它对数据库的影响
          • 保证原子性是数据库系统本身的职责，由 DBMS的事务子系统实现
        • 2. 一致性-【事务提交后，数据库从一个一致性状态变到另一个一致性状态】
          • 一致性状态：数据库中的数据满足完整性约束
          • 如果事务在开始和结束时是使数据库从一个一致性状态转变到另一个一致性状态，那该事务保持了一致性
          • 这个性质由包含事务的应用程序来完成，也可由系统检测完整性约束来自动完成
        • 3. 隔离性-【在事务完成之前，它对数据库产生的结果不能被其它事务引用】
          • 并发执行的事务应该是相对独立的，一个事务的执行不能被其他事务干扰，各个事务的内部操作应该是相互隔离的
          • 隔离性是由DBMS的并发控制子系统实现的
        • 4. 持久性-【一旦事务执行成功(提交)，其对数据库产生的效果永久有效】
          • 持久性也称持续性
          • 已经提交的事务的数据库的改变在任何数据库中应该是持续存在的，是永久的，即便是系统发生故障也不应该对其有任何影响
          • 事务的持久性由DBMS的恢复管理子系统实现
      • 说明
        • 事务管理功能-保证事务的ACID特性
          • 原子性 DBMS的事务子系统
          • 一致性 包含事务的应用程序 or 系统
          • 隔离性 DBMS的并发控制子系统
          • 持久性 DBMS的恢复管理子系统
        • 事务ACID特性可能遭到破坏的原因：
          • 多个事务并行运行时，多个事务中的操作交叉执行；
          • 事务正在执行时，被强迫终止。
        • 数据库的恢复和并发控制需要保证事务的ACID特性
          • 并发控制 需要保证 事务的原子性 一致性 隔离性
          • 恢复 需要保证 事务的原子性 持久性
    • 2. 数据库恢复
      • 概述
        • 从事务的定义和事务的特性来看，当事务提交给DBMS的时候，系统应该保证以下两者之一：
          • 或者成功完成事务中的所有操作，并且这些操作所产生的影响永久记录在数据库中
          • 或者该事务没有对数据库或者其他事务造成影响
        • 数据库恢复
          • 把数据库从错误状态恢复到某一已知的正确状态（即 一致状态或完整状态）的功能
          • BDMS必须具有这项功能
        • 数据库恢复机制是为了保证事务的原子性和持久性
          • DBMS的恢复机制撤销所有未完成事务对数据库的一切影响，保证事务的原子性
          • 对已经提交的事务要恢复它对数据库内数据的更改，保证事务的持久性
      • 2.1 数据库故障的种类
        • 数据库系统中可能发生的故障很多，归纳起来可以分为4类：事务内部故障、系统故障、介质故障、计算机病毒
        • 1. 事务内部故障
          • 预期故障，在程序中可以预先估计到的错误
            • 处理：可以事先在程序的代码中加以判断和添加ROLLBACK语句。当事务执行到ROLLBACK语句时，由系统对事务进行退回操作，即进行UNDO操作
            • 例子-银行转帐业务：从帐号甲将一笔金额转入帐号乙
              • BEGIN TRANSACTION
                • 读取帐号甲的余额BALANCE;
                • BALANCE = BALANCE – AMOUNT;
                • 写回BALANCE；
                • if (BALANCE < 0) then { ROLLBACK; }
                • else {
                  • 读取帐号乙的余额BALANCE1;
                  • BALANCE1 = BALANCE1 + AMOUNT;
                  • 写回BALANCE1；
                  • COMMIT; }
          • 非预期故障，在程序中发生的未估计到的错误
            • 处理：由系统直接执行UNDO处理
            • 非预期的事务内部故障：运算溢出、违约
        • 2. 系统故障
          • 系统故障是指 造成系统停止运转并要求系统重启的事件
          • 原因：
            • CPU故障、操作系统故障、突然断电
            • 系统停机，需要重启。如CPU故障、OS故障、DBMS代码错误、系统断电等
          • 影响正在运行的事务，但不会破坏数据库
            • 发生系统故障后，系统必须重新启动，内存中数据库工作区内数据肯能丢失，但存储在外存储器设备上的数据库数据未遭到破坏
          • 处理：
            • 系统故障导致所有正在运行的事务都以非正常的方式中止，造成数据处于不一致状态
            • 有些数据还保留在内存工作区并为写入数据库，造成数据库处于不一致的状态
            • 需要撤销所有未完成事务，并对所有已提交的事务重做，将数据真正恢复到一致的状态
        • 3. 介质故障
          • 介质故障又叫做硬故障，指外存故障，使得数据库受到破坏
          • 硬故障：磁盘损坏、磁头碰撞、瞬间强磁场干扰等
          • 虽然介质故障发生的几率小，但破坏性最大，属于灾难性故障
          • 处理： 数据库备份
        • 4. 计算机病毒
          • 计算机病毒是认为的故障或者破坏
          • 破坏系统、修改数据
          • 处理：数据库一旦被破坏，仍然要使用恢复技术加以恢复
      • 2.2 数据库恢复的实现技术 #简答
        • 概述
          • 数据库恢复的基本原理是 冗余，即数据的重复存储
          • 数据库中的任何一部分被破坏的或不正确的数据可以根据存储在系统别处的冗余数据来重建
          • 最常用的冗余数据有 后备副本和日志文件
        • 数据库恢复技术涉及两个关键问题
          • 如何建立冗余备份
          • 如何利用这些冗余数据实施数据库恢复
        • 通过数据转储建立冗余
          • 通常在一个数据库系统中利用数据转储和日志文件来建立冗余备份
          • 数据转储
            • 由DBA定期的将整个数据库复制到磁带或另一个磁盘
            • 这个数据库的副本 称为 后备副本 或 后援副本
          • 当数据库发生故障时，可将最近的后备副本重新装入，把数据库恢复起来
            • 但还需要重新运行 转储点之后到故障期间的全部更新事务
          • 静态转储
            • 在系统无运行事务时进行的转储操作
            • 数据库处于一致性状态，转储期间不允许对数据库的任何存取、修改操作
            • 静态转储简单，但必须等待事务结束后佳能进行，新事务必须等转储结束后才能执行
            • 静态转储能保证副本和数据库的一致性，但效率比较低
              • 优点：简单、保证副本和数据库数据的一致性
              • 缺点需等待
          • 动态转储
            • 转储期间允许对数据库进行存取或修改，即转储和用户事务可以并发执行
            • 克服静态转储的缺点，不能保证副本和数据库的一致性
            • 必须把转储期间各个事务对数据库的修改活动记录下来，建立日志文件。后援副本加上日志文件就能把数据库恢复到某一时刻的正确状态
          • 增量转储
            • 每次转储上一次转储后更新过的数据
            • 转储的数据量少，转储时间减少，转出周期缩短，从而可以减少丢失的数据
          • 海量转储
            • 每次转储数据库中的全部数据
            • 可以将增量转储和海量转储结合起来使用
        • 通过日志文件建立冗余
          • 日志文件是用来记录对数据库每一次更新活动的文件，用来记录事务对数据库的更新操作的文件
          • 两种格式的日志文件：
            • （1）以记录为单位的日志文件
            • （2）以数据块为单位的日志文件
          • 每次更新活动的内容作为一个纪录写入“日志”文件，主要内容包括：执行操作的事务标识、操作类型、更新前的旧值、更新后的新值、更新日期和更新时间
          • 日志文件的记录格式：
            • 事务标识 | 操作类型 | 对象标识 | 前像 | 后像
            • 前像 更新前的旧值；后像 更新后的新值
          • 日志文件的作用：
            • 进行事务故障恢复和系统故障恢复，并协助后备副本进行介质故障恢复。
            • 事务故障和系统故障恢复必须用日志文件；
            • 在动态转储方式中必须建立日志文件，并结合后备副本和日志文件对数据库进行有效的恢复；
            • 静态转储时也可建立日志文件
          • 为了保证数据库恢复的正确性，登记日志文件的两条原则
            • 1. 登记的次序必须严格按照并发事务执行的时间次序
            • 2. 必须先日志文件，后写数据库，并且日志文件不能和数据库放在同一磁盘上，要经常把它备份
              • 这条又称为 运行记录优先原则
              • 至少相应运行记录已经写入日志文件后才允许事务向数据库中写记录
              • 直至事务的所有运行记录已经写入日志文件后，才能完成事务的“结束”处理
          • 利用日志文件恢复事务的过程
            • 第一步：从头扫描日志文件，找出在故障发生前已提交的事务（这些事务有BEGIN TRANSACTION 和COMMIT记录），同时找出尚未完成的事务（只有BEGIN TRANSACTION，无COMMIT）
            • 第二步：对尚未结束处理的事务进行撤销（也叫UNDO）处理，对已经结束的事务进行重做（REDO）处理
          • 处理方法
            • 进行UNDO处理的方法
              • 反向扫描日志文件，对每个UNDO事务的更新操作执行反操作，即 插入的删除，删除的重新插入，修改的恢复旧值
            • 进行REDO处理的方法
              • 从头扫描日志文件，依据登录日志文件中次序，重新执行等级的操作
        • 数据库恢复策略
          • 根据数据库的故障类型，数据库恢复要采取不同的策略
          • 事务故障的恢复
            • 事务故障一定是在事务提交前发生，应撤销该事务对数据库的一切更新，
            • 由系统完成，对用户透明
            • 步骤
              • 1. 反向扫描日志文件，查找该事务的更改操作
              • 2. 若查到更新操作，则将日志文件“前像”写入数据库；若查到插入操作，则将数据对象删除；若是删除操作，则将“前像”插入
              • 3. 继续反向扫描日志文件，找出其他的更新操作，并做同样处理，直至遇到该事务的START标记
          • 系统故障的恢复
            • 对已经提交的事务需要重做，对未提交的事务必须撤销所有对数据库的更新
            • 恢复由系统在重启的时候自动完成，不需要用户干预
            • 步骤
              • 1. 从头扫描日志文件，找出在故障发生前已提交的事务（已有COMMIT），将其记入重做（REDO）队列。同时找出尚未完成的事务（只有START无COMMIT或ROLLBACK），将其记入撤销（UNDO）队列
              • 2. 对重做队列中的每个事务进行REDO操作，即 正向扫描日志文件，依据登入日志文件中的次序，重新执行登记的操作
              • 3. 对撤销队列中的每个事务进行UNDO操作，即 反向爱那个扫描日志文件，依据登入日志文件中相反次序，对每个更新操作执行逆操作，从而恢复原状
            • 检查点
              • 为了减少系统故障恢复时扫描日志记录的数目以及重做已提交事务的工作量
              • 检查点记录的内容：
                • 建立检查点时刻正在执行的事务清单
                • 这些事务最近一个日志记录的地址
              • 设置检查点要执行下列操作
                • 暂停事务的执行
                • 将上一个检查点后已提交的事务留在内存工作区，所有更新的数据写入数据库
                • 在日志文件中写入一个检查点记录
              • 在系统运行过程中，DBMS按一定的时间间隔在日志文件中设置一个检查点
              • 在系统故障恢复时，记入重做队列的事务，只是从最近的检查点到发生故障时已经提交的事务，大大减少故障恢复时的工作量
          • 介质故障的恢复
            • 恢复的方法是 重装数据库然后重做已经完成的事务
            • 具体措施：
              • 1. 必要时更换磁盘
              • 2. 修复系统（操作系统和DBMS），重新启动系统
              • 3. 重新装入最近的后备副本
              • 4. 重新装入有关的日志文件副本，重做转储最近后备副本以后提交的所有事务
    • 3. 并发控制
      • 概述
        • 作为共享资源的数据库，可以同时供多个用户使用，同一时刻并行运算的事务数目可达到好多个
        • 事务应该具有一致性，如果多个事物依次顺序执行，一个事务完全结束后，另一个再开始执行，这种执行方式称为串行执行。
          • 显然，事务的串行执行可以保证数据库的一致性
        • 为了充分利用系统资源，又能改善对事务的响应时间，应该允许多个事务在时间上交叉执行，并发地存取数据库，这种方式称为 并发存取
      • 并发操作引发的问题
        • 冲突
          • 不同事务的两个操作均是针对同一个数据对象，至少一个是写操作，则称这两个事务是冲突的
            • 读-写冲突、写-读冲突、写-写冲突
          • 并发操作破坏了事务的隔离性
            • 如果对并发事务中的操作不加以控制，可能会引发 丢失修改、读“脏数据”、不可重复读
            • 冲突操作破坏了事务的隔离性
        • 丢失修改
          • 两个事务都对某一数据对象读取，修改并写入数据库，导致先写入的事务修改被覆盖而丢失
            • 两个事务T1和T2读入同一数据并修改，T2的提交结果破坏了T1提交的结果，导致T1的修改被丢失
          • 由 写-写冲突 引起
        • 读 “脏数据”
          • 因为某种原因T1中途失败，对A的修改通过ROLLBACK撤销，于是A又恢复成原值，这样，T2读取的数据就不正确了
          • 这是由 写-读 冲突引起
        • 不可重复读
          • 一个事务如果没有更新数据库数据的操作发生，则同一个查询操作执行两次或多次的结果应该是一致，如果不一致，就说明产生了不可重复读的现象
          • 不可重复读是指事务T1读取数据后，事务T2执行更新操作，使T1无法再现前一次读取结果。
          • 不可重复读包括三种情况：
            • (1)事务T1读取某一数据后，事务T2对其做了修改，当事务T1再次读该数据时，得到与前一次不同的值
            • (2)事务T1按一定条件从数据库中读取了某些数据记录后，事务T2删除了其中部分记录，当T1再次按相同条件读取数据时，发现某些记录消失了
            • (3)事务T1按一定条件从数据库中读取某些数据记录后，事务T2插入了一些记录，当T1再次按相同条件读取数据时，发现多了一些记录。
            • 后两种不可重复读有时也称为幻影现象（Phantom Row）
          • 这是有 读-写冲突引起
        • 三种冲突的比较
        • 并发控制的任务
          • 通过合理的调度方式来控制并发事务的正确执行，使得多个事务之间互不干扰，避免造成数据库中数据的不一致
      • 调度及其可串行化
        • 如果两个调度S1和S2在数据库任何状态下，对于相同的初始状态，执行结构都是一样的，称 S1和S2等价
        • 串行调度：
          • 如果一个调度中，各个事务没有交叉执行，而是串行执行，称为串行调度
          • 不同次序会得到不同结果，但保持数据的一致性，都认为是正确的
          • 由于串行调度中，一个事务运行过程中没有其他事务同时运行，不会受到其他事务的干扰，因此，串行调度的结果总是正确的
        • 并发调度
          • 在一个调度中，各个事务交叉地执行，这个调度称为 并发调度
        • 可串行化调度
          • 当一个事务集的并发调度和某一串行调度是等价的，称该并发调度为 可串行化调度
          • 可串行化调度保持了数据库的一致性，是正确的
          • 一般的DBMS中，可串行化作为并发调度正确与否的判定准则
          • 造成调度不等价的主要原因 执行次序不同而引起
          • 可串行化的判断-可用 前趋图 来判断
            • 算法

        • 为了保证并发调度是可串行化的，目前DBMS普遍采取的是封锁技术。
      • 封锁技术
        • 封锁
          • 封锁是实现并发控制的一种机制
          • 事务T在对数据进行操作之前，先向系统申请对其进行加锁，获得锁后不允许其它事务更新该数据，直到事务T释放该数据上的锁。
          • 封锁技术中，主要有两个锁：
            • 排它锁 和 共享锁
        • 排它锁
          • X锁，写锁
          • 若事务T对数据对象A加上X锁，则只允许T读取和修改A，不允许其他任何事务再对A加锁，直到T释放A上的X锁
          • 加了X锁，不能再加任何锁
        • 共享锁
          • S锁，读锁
          • 对于同一数据对象允许多个事务并发地读，但对写操作是排他的
          • 加了S锁，其他事务还能在加S锁，但不能加X锁
        • 两种锁的关系
          • [锁相容矩阵 p181]
        • 封锁协议
          • 封锁协议：
            • 在运用X锁和S锁时，还需要一些约定规则，如何时加锁、持续时间、何时释放，这些规则称为 封锁协议
          • 三级封锁协议：
            • 三级封锁协议分别在不同程度上解决了对并发操作的不正确调度可能会带来的丢失修改、不可重复读、读脏数据等不一致问题，为并发操作的正确调度提供了一定的保证
            • 不同级别的封锁协议达到的系统一致性级别不同
              • [p183 不同级别的封锁]
            • 1. 一级封锁协议
              • 一级封锁协议
                • 事务T在修改数据R之前必须先对其加X锁，直到事务结束才释放
                • 事务结束包括 正常结束（COMMIT）和非正常结束（ROLLBACK）
              • 一级封锁协议可防止 丢失修改，并保证事务T是可恢复的
              • 不能保证 可重复读 和 不读“脏”数据
                •  在一级封锁协议中，如果仅仅是读数据不对其进行修改，是不需要加锁的
            • 2. 二级封锁协议
              • 二级封锁协议
                • 一级封锁协议加上事务T在读取数据R之前必须先对其加S锁，读完后方可释放S锁。
              • 二级封锁协议除防止了丢失修改，还可以进一步防止读“脏”数据。
              • 在二级封锁协议中，由于读完数据后即可释放S锁，所以它不能保证可重复读。
            • 3. 三级封锁协议
              • 三级封锁协议
                • 一级封锁协议加上事务T在读取数据R之前必须先对其加S锁，直到事务结束才释放。
              • 三级封锁协议除防止了丢失修改和不读“脏”数据外，还进一步防止了不可重复读。

            • 4. 两段锁协议（2PL）
              • 两段锁协议是保证调度可串行化的基本封锁技术的协议
              • 规定
                • 在任何数据进行读写操作之前，事务要获得对数据对象的加锁
                • 在释放一个锁之后，事务不在获得任何其他锁
                • Note
                  • 一个事务的所有加锁操作应在第一个解锁操作之前
                  • 规则1避免了两个冲突操作同时存取同一数据。
                  • 规则2把每个事务分为两个阶段：上升段和下降段。
              • 若并发执行的所以事务均遵守两段锁协议，则对这些事务的所以调度都是可串行化的
                • 事务遵守2PL协议是可串行化调度的充分条件，而不是必要条件。
                • 即若并发事务都遵守2PL协议，则对这些事务的任何并发调度策略都是可串行化的；
                • 若对并发事务的一个调度是可串行化的，不一定所有事务都符合2PL协议。
              • 2PL是调度可串行化的充分条件，并不是必要条件
              • 所谓“两段”锁的含义是，事务分为两个阶段，第一阶段是获得封锁，也称为扩展阶段，第二阶段是释放封锁，也称为收缩阶段
              • 如果遵守2PL的调度，将锁的释放都放在事务结束，则可以避免 连锁回滚 和 不可重复读 这些问题
              • 严格的两段锁协议
                • 把所有锁都放在事务结束时释放
              • 优点：简单
              • 缺点：易死锁
                • 例如：对于如下两个事务采用两段锁协议
                  • T1：R1(X) W1(Y)
                  • T2：W2(Y)W2(X)
                  • T1与T2的一个调度过程：
                  • 1 初始时，没有事务占有锁
                  • 2 调度器接到R1(X) ， 对X加读锁，执行R1(X) 。
                  • 3 调度器接到W2(Y) ，对Y加写锁，执行W2(Y)
                  • 4 调度器接到W2(X) ，T2等待
                  • 5调度器接到W1(Y) ， T1等待
                • 这样就造成了死锁。
      • 封锁导致的问题
        • 死锁
          • 系统中有两个或两个以上的事务处于等待状态，并且每个事务都在等待其中另一个事务解除封锁，它才能够执行下去，结果造成任何一个事务都无法继续执行
          • 事务T1已经封锁A，而又想申请封锁B，而此时事务T2已经封锁B，而又想申请封锁A，这样，T1等待T2释放B，而T2等待T1释放A，使得T1、T2均无法继续执行下去，这种情况称为死锁。
          • 解决方法-预防死锁 和 死锁检测，发现死锁后解除死锁
          • 预防死锁
            • 一次封锁法
              • 要求每个事务必须一次将所有要使用的数据全部封锁，否则，不能执行。
              • 缺点：降低了系统的并发度。
            • 顺序封锁法
              • 预先对数据对象规定一个封锁次序，所有事务都按这个次序实行封锁。
              • 缺点：排列封锁次序困难。
          • 死锁检测
            • 超时法：
              • 如果一个事务等待的时间超过某个时限，认为发生死锁
              • 超时法容易实现，但时间长短难以确定
            • 等待图法
              • 当等待图出现回路时，存在死锁
              • 发现死锁后，由所管理器做如下处理：
                • 在循环等待事物中选择一个牺牲代价最小的事务执行回滚，并释放它获得的锁和其他资源，使得其他事务得以运行下去
        • 活锁
          • 系统可能使得某个事务永远处于等待状态，得不到解锁的机会
          • 事务T1，T2申请数据对象A，T1先给A加锁，T1释放A上的锁后，事务T3又给A加锁，T2等待，这样，A始终被其他事务封锁，事务T2可能长时间得不到A，这种情况称为活锁。
          • 避免活锁的方法就是 采用先来先服务的策略
            • 当所个事务请求封锁同一数据对象时，封锁子系统按请求封锁的先后顺序对这些事务排队，该数据对象上的锁一旦释放，首先批准申请队列中第一个事务获得锁
      • 封锁的粒度
        • 封锁数据对象的大小称为 粒度
          • 封锁的数据对象可以是 逻辑单元，也可以是 物理单元
          • 封锁的数据对象可以很粗，也可以很细
        • 粒度越粗，封锁处理越简单，系统开销越小，但这样往往把不需要加锁的数据也封锁了，降低了部分的并发度
        • 粒度越细，往往要加很多的锁，系统开销较大，提高了系统的并发度
    • 4. 数据库安全性
      • 数据库安全性
        • 数据库的安全性:就是防止非法用户使用数据库造成数据泄露、更改或破坏，以达到保护数据库的目的
        • 保护数据库以防止不合法使用
        • 计算机系统安全性
          • 为计算机系统建立和采取的各种安全保护措施，保护计算机系统（硬件、软件及数据）不会遭到破坏。
        • 一般计算机系统中，安全措施是一级一级层层设置的
      • 4.1 用户标识和鉴别
        • 用户标识域鉴别是系统提供的最外层安全保护措施
        • 在DBMS中，DBA可以通过为数据库用户创建用户帐户和密码
          • 每次需要对数据库进行存取的时候，用户都必须通过输入帐号和密码才能登录
          • 每次用户要求进入系统时，由系统进行核对，通过鉴定帐号和密码正确以后才提供数据库使用权
        • 常用的方法：通过用户帐户和密码来鉴定用户身份的合法性
          • 简单易行，但容易被比别人窃取和破坏
        • 用户标识：Identification
          • 用户名：User Name，或者
          • 用户标识号：UID
        • 用户鉴别：Authentication
          • 用户名+口令：Password
      • 4.2 存取控制
        • 数据库安全的主要思路：存取控制
          • 给合适的人以合适的权限
        • 存取控制机制
          • 定义用户及其权限，在数据字典中登记用户权限；
          • 权限合法性检查
        • 存取控制
          • 确保具有授权资格的用户访问数据库的权限，同时使得所有未授权的人员无法访问数据库的机制
        • 数据库用户一般可以分为三类：
          • 一般数据库用户
            • 通过授权可以对数据库进行操作的用户
          • 数据库拥有者
            • 数据库的创建者
            • 除一般用户拥有的权限外，还可以授予或回收其他用户对其所创建的数据库的存取权
          • 有DBA权限的用户
            • 用户支配整个数据库资源的权限，对数据库用于最大的特权，因而也对数据库负有特别的责任
            • 通常只有数据库管理员（DBA）有这种特权
            • DBA特权命令包括
              • 给各个独立的账户、用户或者用户组授予特权和回收特权
              • 把某个适当的安全分类级别指派给某个用户
        • 授权
          • 为了保证用户只能访问自己有权限的数据，必须对每个用户授予不同的数据库存储权限，称为授权
          • 第一个拥有DBA权限的用户是由系统设置的，在系统初始化时，系统中至少有一个具有DBA特权的用户，密码也是系统规定好的
        • 大型数据库管理系统都支持自主存取控制，在SQL中，授权语句有两种：授予权限语句和授予角色语句
          • 两类实现存取控制的方法
            • 自主存取控制: Discretionary Access Control
              • a)用户对不同的数据对象有不同的权限；
              • b)不同用户对同一对象具有不同的权限；
              • c)允许用户之间的传递授予权限
            • 强制存取控制: Mandatory Access Control
              • a)以一定的密级标注数据对象
              • b)授予每个用户以某一级别的许可证
          • 自主存取控制DAC
            • SQL标准对DAC的支持
              • 授权：Grant语句
              • 回收：Revoke语句
            • SQL创建用户时赋予用户的权限
              • DBA、RESOURCE、CONNECT
              • Create User <用户名> [With DBA | RESOURCE | CONNECT]
              • 默认用户权限： CONNECT
              • CONNECT用户的初始权限：只能登录数据库
              • RESOURCE用户的初始权限：
                • 创建基本表和视图，并成为所创建对象的属主
                • 不能创建模式和新用户
                • 可授权
              • DBA用户的初始权限：
                • 创建新用户、模式、基本表、视图等
                • 拥有对所有数据库对象的存取权限
                • 可授权
            • 授权权限：Grant语句
              • 把“在某个对象上” “干某种事情”的权限授予“某个用户”
              • GRANT  <权限> ON <数据对象> TO <用户> [WITH GRANT OPTION]
            • 回收权限：Revoke语句
              • 把“在某个对象上” “干某种事情”的权限从“某个用户”那里要回来
              • REVOKE <权限> ON <数据对象> FROM <用户> [CASCADE | RESTRICT]
                • REVOKE UPDATE(Sno) ON TABLE Student From user4;
                • 可以是具体到某个属性
            • PUBLIC 是所有数据库用户的总称
            • WITH GRANT OPTION 授权者可以将此特权转授给其他用户
            • 所有操作权限 ALL PRIVILEGES
            • 授权语句的例子
              • GRANT SELECT ON TABLE 表1 TO  PUBLIC；
                • //将对表1的SELECT特权授予所有的用户。
              • GRANT SELECT（列1，列5） ON 表1 TO sam；
                • //将对表1中1和5列的INSERT特权授予sam。
              • GRANT ALL PRIVILEGES ON TABLE 表1 TO bob，sam；
                • //将对表1的所有操作特权授予bob和sam。
              • GRANT UPDATE（列2，列4） ON 表1 TO wang WITH GRANT OPTION；
                • //将对表1中2和4列的UPDATE特权授予wang，并允许他将此特权转授给其它用户
            • 以单个权限为单位授权太过麻烦
            • 角色：权限的集合
              • 为具有相同权限的用户群创建一个角色
            • 基于角色来管理权限简化授权过程
              • 创建角色
              • 给角色授权或授予角色
              • 给用户授予角色
            • 角色的创建
              • CREATE ROLE <角色名>
            • 给角色授权
              • GRANT <权限> ON <数据对象> TO <角色>
              • GRANT <角色> TO <角色> [WITH ADMIN OPTION]
            • 给用户授予角色
              • GRANT <角色> TO <用户> [WITH ADMIN OPTION]
            • 角色权限的回收
              • REVOKE  <权限>   ON  <数据对象>   FROM  <角色>
        • 在DBMS的数据字典中为每个数据库设置了一张授权表，存放了每个授权ID的授权定义。
          • 授权ID | 数据对象 | 存取权限
        • 当用户发出对数据库存取操作请求时，DBMS在数据字典中查询授权表，对操作的合法性进行检查
      • 4.3 视图机制
        • 通过为不同的用户定义不同的视图，可以将要保护的数据对无权存取的用户隐藏，从而自动地给数据提供一定程度的安全保护
        • 例如，给某用户定义了一个只读视图，并且这个视图的数据来源于关系R，则此用户只能读R中的有关信息，数据库中一切其他信息对他都是隐藏的。
        • 例子
          • 王平老师只能查询计算机系学生的信息
          • 系主任李明具有处置计算机系学生信息的所有权限
          • CREATE VIEW CS_STU AS Select * From Student Where Sdept = ‘CS’;
          • GRANT SELECT ON CS_STU TO 王平;
          • GRANT ALL PRIVILIGES ON CS_STU TO 李明
      • 4.4 数据加密
        • 加密技术是防止数据库中数据在存储或者传输中泄露的有效手段。
        • 加密的基本思想-根据一定的算法将原始数据(明文)变换成不可直接识别的格式(密文)，从而使得不知道解密算法的人无法获知数据的内容。
        • 由于加密和解密 费时而且占用大量系统资源，增加开销，降低数据库的性能
      • 4.5 审计
        • 审计：C2级DBMS的必要指标，是把用户对数据库的所有操作自动记录下来放入审计日志
          • 用户级审计
          • 系统级审计
        • 审计功能把用户对数据库的操作自动记录下来放入审计日志。
        • 系统能利用这些审计跟踪的信息，重现导致数据库现状的一些列事件，以找出非法存取数据的人
        • 审计通常是很费时间和空间的，所以DBMS往往将其作为可选的，允许DBA和数据的拥有者选择。
        • 数据库审计对于被多个事务和用户更新的敏感型数据库是非常重要的。
      • 没有哪个安全措施是绝对可靠的
        • 安全性保护措施越复杂、越全面，系统的开销就越大，用户使用就越困难
    • 5. 数据库完整性
      • 概述
        • 数据的完整性是指 数据库中数据的正确性和相容性
          • 即为了防止数据库中存在不符合语义的数据，防止错误信息的输入和输出
        • 为了实现数据库完整性，DBMS必须提供表达完整性约束的方法，以及实现完整性的控制机制
        • DBMS对数据库完整性的支持
          • 提供定义完整性约束的机制
            • 实体完整性、参照完整性、自定义完整性
          • 提供完整性检查的方法
          • 违约处理：拒绝执行、级联执行
      • 5.1 完整性约束条件的类型
        • 数据库的完整性
          • 指数据库中的数据的正确性和完整性
          • 即为了防止数据库中存在不符合语义的数据，防止错误信息的输入和输出
          • 这些约束条件必须存放在数据字典中
        • 完整性约束：加在数据之上的语义约束
        • 完整性约束条件的作用对象可以是 列、元组、关系 3种
          • 关系：若干元组间、关系集合上以及关系之间的联系的约束
          • 元组：元组的约束是元组中各个字段间的联系的约束
          • 属性列：主要是列的数据类型、取值范围、精度、排序等约束条件
        • 完整性约束条件的类型可以分为两大类：静态约束和动态约束
        • 静态约束
          • 静态约束
            • 数据库每一确定状态时的数据对象所应满足的约束条件
            • 是反应数据库状态合理性的约束
            • 是最重要的一类完整性约束
          • 按约束对象的不同，分为三类：静态列约束、静态元组约束、静态关系约束
            • 静态列约束是对一个列的取值域等的说明和限制，包括：
              • 对 数据类型 的约束
              • 对 数据格式 的约束
              • 对 取值范围 的约束
              • 对 空值 的约束
              • 其他约束
            • 静态元组约束
              • 一个元组是由若干个列值组成，静态元组约束规定组成一个元组的各个列之间的约束关系
              • 例如：库存关系中出库数量不能大于库存数量。
            • 静态关系约束
              • 反映了一个关系中各个元组之间或者若干关系之间存在的关系或者约束，包括
                • 实体完整性约束
                • 参照完整性约束
                • 函数依赖约束
                • 统计约束，即某个字段值域一个关系多个元组的统计值之间的约束关系
          •  实体完整性约束； 参照完整性约束；
        • 动态约束
          • 动态约束是指数据库从一种状态转变到另一种状态时，新旧值之间所满足的约束条件
          • 反映了数据库状态改变时应遵守的约束
          • 按约束对象的不同，分为三类
            • 1. 动态列约束：
              • 动态列约束规定修改列定义或列值时应满足的约束条件
              • 修改列定义时的约束
              • 修改列值时的约束
            • 2. 动态元组约束
              • 动态元组约束是指修改某个元组的值时元组的各个字段之间要满足某种约束条件
            • 3. 动态关系约束
              • 动态关系约束是加在关系变化时的约束条件
          • 事务的ACID特性：原子性/一致性/隔离性/持久性
      • 5.2 完整性控制机制的功能
        • 定义功能
          • 提供定义完整性约束条件的机制
        • 检查功能
          • 检查用户发出的操作请求是否违背了完整性约束
          • 立即执行的约束
            • 检查是否违背完整性约束条件通常是在一条语句执行后立即检查，称这类约束为立即执行的约束
          • 延迟执行的约束
            • 完整性检查需要延迟到整个事务结束后再进行，称这类约束为延迟执行的约束
        • 保护功能
          • 违背了完整性约束，采取一定的保护动作来保证数据的完整性
          • 最简单的保护数据完整性的动作就是拒绝该操作，但也可以采取其他处理方法。
          • 如果发现用户操作请求违背了延迟执行的约束，由于不知道是哪个或哪些操作破坏了完整性，所以只能拒绝整个事务，把数据库恢复到该事务执行前的状态。
      • 5.3 完整性约束的表达方式 #略
        • 在创建和修改基本表模式时说明约束
          • SQL中的关键字
            • PRIMARY KEY
              • 两种使用方法
              • 只能出现一次
            • UNIQUE
              • 可以出现多次
          • 参照完整性和外部关键字
            • 两种方法
              • 直接加 REFERENCES ...(表名.属性)
              • 末尾 FOEIGN KEY 属性 REFERENCES ...(表名.属性)
            • 维护策略
              • 默认策略：拒绝违规的更新
              • 级联策略：被参照对象修改了，参照对象对应的也被自动修改
              • 置空策略：（不考）
          • 关于属性值的约束
          • 全局约束
        • 用断言说明约束
        • 用触发器表示约束
        • 用过程说明约束
        • 【实体/参照完整性约束】
          • 实体完整性：PRIMARY KEY
            • 列级定义
            • 关系级定义
          • 参照完整性：
            • FOREIGN KEY / REFERENCES
            • 违约处理：拒绝、级联删除、设置为空
          • 例子
            • CREATE TABLE SC (
            • Sno CHAR(8) NOT NULL,
            • Cno CHAR(4) NOT NULL,
            • Grade SMALLINT,
            • PRIMARY KEY (Sno, Cno),
            • FOREIGN KEY (Sno) REFERENCES Student(Sno)
            • ON DELETE CASCADE
            • ON UPDATE CASCADE,
            • FOREIGN KEY (Cno) REFERENCES Course(Cno)
            • ON DELETE NO ACTION
            • ON UPDATE CASCADE
            • );
        • 【用户自定义完整性约束】
          • 定义属性列的约束条件
            • 列值非空：NOT NULL
            • 列值唯一：UNIQUE
            • 语义检查：CHECK短语
          • 举例：
            • 性别只允许取值“男”或“女”
              • CREATE TABLE Student (
              • … …
              • Ssex CHAR(2) CHECK (Ssex IN (“男”, “女”)),
              • … … );
            • 成绩只能在0-100之间:
              • CREATE TABLE SC (
              • …. ….
              • Grade SMALLINT CHECK (Grade>=0
              • and Grade<=100),
              • … …);
          • 违约处理和保护：
            • create table s(
            • sno char(2) primary key,
            • sgender char(2) check(sgender in('f','m')))
          • 定义元组的约束条件：CHECK
            • 男性的名字一定不能以Ms. 打头
            • CREATE TABLE Student (
            • … …
            • Sname CHAR(8) NOT NULL,
            • Ssex CHAR(2) CHECK (Ssex IN (‘男’,‘女’)
            • and (Ssex=‘女’) OR Sname
            • NOT LIKE ‘Ms. %’)),
            • … … );
          • 完整性约束命名子句
            • CONSTRAINT <约束名> [PRIMARY KEY短语 | FOREIGN KEY短语 | CHECK短语]
            • CREATE TABLE Student (
            • Sno NUMERIC(6) CONSTRAINT c1
            • CHECK (Sno BETWEEN 90000 AND 99999),
            • Sname CHAR(8)  CONSTRAINT c2 NOT NULL,
            • Ssex CHAR(2) CONSTRAINT c3
            • CHECK (Ssex IN (‘男’,‘女’)),
            • … …
            • );
          • 域约束：DOMAIN
            • 建立性别域：
            • CREATE DOMAIN GenderDom CHAR(2) CHECK (VALUE IN (‘男’, ‘女’));
            • CREATE TABLE Student (
            • Ssex GenderDom ,
            • … …
            • );
          • 触发器：TRIGGER
            • 定义在关系表上的由事件驱动的特殊过程
            • 一旦定义，任何用户对表的增删改操作均会由DBMS自动激活相应的触发器，进行完整性检查
            • 非SQL核心规范的内容，但一般DBMS均提供支持
            • 例子
              • 创建一个触发器sc_trig，当插入或者更新“学习”表时，该触发器检查“成绩”是否在指定的范围内
              • CREATE TRIGGER [sc_trig] ON [dbo].[学习]
              • FOR INSERT, UPDATE
              • AS
              • declare @score smallint
              • select @score=score
              • from   学习
              • if (@score<0 ) or ( @score >100)
              • begin
              • raiserror('学成成绩必须在0到100之间',16,1)
              • rollback transaction
              • end
